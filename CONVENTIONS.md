## 0. 이 프로젝트의 목표

**5일 안에 완벽한 제품 ❌ / 5일 안에 남이 쓸 수 있는 URL ⭕**

기능을 추가할지 말지 고민되면 **빼는 쪽**을 고른다.

### 되돌릴 수 없는 일정 결정

> **D3에 무조건 웹으로 나간다.** 파이프라인 상태와 무관하다.

이건 태스크 계획이 아니라 **되돌릴 수 없는 결정**이다. 6장의 위험 신호("D4인데 배포 안 됨")는
이미 늦었을 때의 대응이지 예방이 아니다.

**가장 그럴듯한 실패 모드는 이것이다:** D2에 프롬프트 튜닝이 재밌어진다. 강등률이 12%에서 8%로
떨어지는 걸 보면 한 번만 더 고치고 싶어진다. 그렇게 D3와 D4가 통째로 튜닝에 들어간다.
**D4에 웹을 시작하면 유저를 넣을 시간이 안 남는다. 유저 0명이면 이 프로젝트가 증명하려던 것의 절반이 사라진다.**

프롬프트가 아무리 아쉬워도 D3 아침에 튜닝을 중단한다. 갭 정확도 80%짜리 웹이,
정확도 95%짜리 CLI보다 이 프로젝트의 목표에 가깝다.

### D2 종료 기준 — "멀쩡해 보임"은 기준이 아니다

**샘플 1건이 잘 나왔다고 D3로 넘어가지 않는다. 멀쩡해 보이는 것이 가장 위험하다.**

D2를 끝내려면 **서로 다른 공고 최소 10건**을 돌려 강등률 **분포**를 봐야 한다.

- 3건은 표본이 아니다. 1건은 일화다.
- 반드시 **서로 다른 공고**여야 한다.
- 프롬프트를 고치기 **전에** 먼저 10건 베이스라인을 찍는다. 비교 대상이 없으면 개선을 주장할 수 없다.
- 분포는 `scripts/summarize_runs.py`가 `(model, prompt_hash)`별로 뽑는다.

1건 보고 넘어가면, D3에 웹을 띄운 다음 실제 공고에서 무너진다. 그때는 고칠 시간이 없다.

#### ⚠️ 폐기된 전제: "`temperature=0`이면 재현된다"

**D1에 이렇게 적었다가 D2에 실측으로 깨졌다. 기록으로 남긴다.**

> ~~`temperature=0`이므로 같은 입력을 10번 돌리는 것은 무의미하다. 같은 점 10개일 뿐이다.~~

**틀렸다.** `temperature=0`은 재현성을 보장하지 않는다. 같은 입력·같은 `prompt_hash`로 돌린
두 실행에서 quote 제시가 3개 → 4개로 달라졌다. OpenAI API는 temperature 0에서도 결정론적이지 않다.

`seed`는 **넣지 않는다.** best-effort라서 의존할 수 없는데 의존하고 싶게 만든다.
"seed 넣었으니 재현되겠지"는 방금 깨진 "temperature=0이니 재현되겠지"와 같은 종류의 믿음이다.
그리고 seed를 넣든 말든 반복 측정은 어차피 해야 하므로, seed는 아무것도 더 주지 않는다.
**노이즈를 숨기지 말고 측정한다.** 숨기면 잘못된 결론을 자신 있게 내리게 된다.

#### 최소 검출 가능 효과 (MDE)

`scripts/measure_noise.py`로 코드 동결 상태에서 반복 측정해 지표의 산포를 잰다.
**그 산포보다 작은 변화는 '개선했다'고 주장하지 않는다.**

D2 실측 (job1 × 5회, gpt-4o-mini):

| 구간 | 발견율 밴드 |
|---|---|
| 전체 (Step1+Step2) | 0%p (5회 전부 동일 — 운이 좋았을 뿐) |
| Step1 동결 + Step2만 | **9%p** (5회 중 1회가 근거 하나를 놓침) |

- **노이즈원은 Step2다.** Step1은 요구사항 11개를 8회 연속 동일하게 뽑았다.
  → 튜닝 중 **Step1 출력을 동결할 필요 없다.**
- **양자화 한계가 노이즈와 같은 크기다.** 요구사항이 11개라 지표는 `1/11 = 9%p` 단위로만 움직인다.
  "발견율 3%p 개선"은 애초에 표현 불가능한 숫자다.
- **그래서 해법은 반복이 아니라 공고 수다.** 10건이면 요구사항 ~110개 → 눈금이 0.9%p로 잘게 쪼개지고
  노이즈도 평균에서 희석된다. **공고 1건으로는 프롬프트 개선을 영원히 검출할 수 없다.**

**이 숫자들은 job1 한 건에서 잰 값이다. 일반화하지 마라.**
산문형 공고나 영어 공고에서는 Step2가 더 흔들릴 수 있고, Step1도 job1에서만 안정적일 수 있다.
10건 베이스라인을 뜨면 공고별 산포와 Step1 지문 안정성이 **공짜로 딸려 나온다**(어차피 출력이 저장된다).
그때 9%p와 "Step1은 안정적이다"를 **갱신하라. 믿음으로 승격시키지 마라.**

#### 모델 결정 규칙 — 표를 보기 전에 박는다 ⚠️

**결과를 보고 나서 기준을 만들면, 마음에 드는 모델에 맞춰 기준이 휜다.**
낙관적 자동 판정 로직을 코드에서 제거한 것과 같은 원리다. 사후 합리화의 문을 여기서도 닫는다.

적용 순서대로:

1. **지어내기율이 유의미하게 높은 모델은 탈락.** 발견율이 아무리 높아도 탈락이다.
   이 도구의 존재 이유가 근거의 신뢰성이다. 근거를 못 믿으면 나머지는 의미가 없다.
2. **Step1 텍스트 지문이 흔들리는 모델은 감점.** `requirements_count`는 발견율의 **분모**다.
   11→14로 튀면 아무것도 안 바뀌었는데 지표가 ~20%p 움직인다 — 측정하려는 축 자체가 오염된다.
   또한 캐스케이드가 되살아나 튜닝 중 Step1 동결이 강제된다.
3. **지어내기율·발견율이 노이즈 밴드 안에서 구분되지 않으면 → 품질 무승부 선언.**
   그 경우 **지연과 비용으로 결정한다.** "느낌상 더 좋아 보인다"로 결정하지 않는다.

**D2 실측 (job1, Step1 텍스트 지문 대조):**

| 모델 | 요구사항 수 (반복) | 텍스트 지문 |
|---|---|---|
| **gpt-4o-mini** | 11 × 8회 | ✅ 8/8 완전 동일 |
| gpt-5.4-mini | 12, 12, **14** | ❌ 흔들림 |
| gpt-5.4-nano | 11, 11, **14**, 11, 11 | ❌ 흔들림 (`kind`도 기술↔경험 진동) |

두 5.4 모델은 가끔 공고의 "이런 일을 합니다"(= 업무 내용)를 **필수 요구사항으로** 끌어온다.
원문에 실재하므로 지어내기(1번)는 아니지만, **2번 감점에 해당한다.**

> **참고 — `gpt-5.4-mini`의 12번째 항목은 지어낸 것이 아니다.**
> 공고의 "주니어 엔지니어를 멘토링**하거나** 기술 문서를 작성해본 경험"을 둘로 쪼갠 것이다.
> Step1 프롬프트가 "뭉뚱그리지 말고 쪼개라"고 지시하므로 **오히려 프롬프트를 더 잘 지켰다.**
> 쪼갬은 중립이다. 감점은 어디까지나 **회차마다 결과가 달라지는 것**이다.

#### 프롬프트 A/B — 집계 델타 하나만 보지 않는다

10건 합산에서 +2%p가 나와도, 그게 **한 공고에서 +3이 튄 것**인지 **열 건 중 여덟 건에서 고르게 오른 것**인지는
완전히 다른 이야기다. 집계 델타만 보면 전자를 개선이라고 착각한다.

**같은 10건에 프롬프트 A/B를 쌍으로 돌리고, 개선 방향이 일치한 공고가 몇 건인지 센다.**

- **8/10이 같은 방향** → 신호. 개선이라고 주장할 수 있다.
- **5/5로 갈림** → 집계 델타가 아무리 커도 노이즈다.

통계 패키지 필요 없다. **부호만 세면 된다.**

#### 측정은 실제 공고로, 샘플은 시연용

**`data/samples/`의 가상 공고로 분포를 재지 않는다.** 가상 공고는 프롬프트를 쓴 사람이 같이 썼기 때문에,
프롬프트가 잘 처리하는 형태로 무의식적으로 편향된다 — 깔끔한 자격요건/우대사항 구조, 명확한 불릿, 애매한 표현 없음.
**자기가 낸 시험을 자기가 푸는 것이다.** 실제 공고는 회사 소개가 절반이고 요구사항이 문단에 섞여 있다.

| 용도 | 위치 | 커밋 |
|---|---|---|
| **분포 측정** (D2 종료 판단) | `data/private/` 실제 공고 10건 | ❌ |
| **시연/README** | `data/samples/` 가상 공고 3건 | ✅ |

실제 공고로 돌린 결과는 `out/private/`로 가고 커밋되지 않는다(본문이 통째로 들어 있다).
그런데 "프롬프트 커밋과 결과가 짝을 이뤄야 한다"는 규칙이 깨지면 안 되므로, **계측치만 뽑아 커밋한다:**

```bash
uv run python scripts/summarize_runs.py out/private --export   # → out/metrics.jsonl (커밋됨)
```

`summary` 블록에는 텍스트가 없다(모델/해시/개수/지연/토큰/비용뿐). 공고 파일명은 익명 id로 바뀐다.
**내용은 비공개, 측정 증거는 공개.**

## 1. 절대 규칙

### 보안
- `.env`는 **절대** 커밋하지 않는다. `.gitignore` 최상단에 있어야 한다.
- API 키를 코드, 로그, 주석, 에러 메시지 어디에도 넣지 않는다.
- 레포는 처음부터 **public**. (private → public 전환 시 히스토리에 남은 키가 노출된다)
- **실제 이력서/공고는 `data/private/`에만 둔다.** 샘플과 섞이는 순간 사고 난다.
  샘플(`data/samples/`)에는 실명·연락처·실존 회사명을 넣지 않는다.

### LLM 출력은 믿지 않는다
- LLM이 반환한 인용문(`quote`)은 **반드시 원문 대조 검증**을 거친다.
- 검증 실패 시 조용히 넘어가지 말고 **강등(`status="없음"`)하고 카운트한다.**
- 이 카운트는 최종 출력에 항상 표시한다.

### 조용한 실패 금지
- 입력이 너무 길면 **자동으로 자르지 말고 에러**를 낸다.
- 개수 제약(요구사항 8~15개, bullet 2개)을 벗어나면 **경고를 찍는다.** 조용히 잘라내지 않는다.
- **조용한 품질 저하가 가장 나쁘다.** 이상하면 시끄럽게 알린다.

## 2. 코드 스타일 — Ruff로 고정

**사람이 스타일을 논의하지 않는다. 도구가 강제한다.**

### 설정 (`pyproject.toml`)

```toml
[tool.ruff]
line-length = 100
target-version = "py312"   # requires-python(>=3.12), .python-version과 반드시 일치시킬 것

[tool.ruff.lint]
select = [
    "E",    # pycodestyle
    "F",    # pyflakes (미사용 변수/import)
    "I",    # isort (import 정렬)
    "UP",   # pyupgrade (구식 문법 현대화)
    "B",    # bugbear (흔한 버그 패턴)
    "SIM",  # simplify (불필요한 복잡도)
]
ignore = ["E501"]   # line-too-long은 formatter가 처리

[tool.ruff.format]
quote-style = "double"
```

### 명령어

```bash
uv run ruff format .        # 포매팅 (Black 대체)
uv run ruff check . --fix   # 린트 + 자동 수정
```

### 규칙
- **커밋 전에 위 두 명령을 반드시 실행한다.**
- Ruff가 잡은 건 **논쟁하지 않고 고친다.**
- `# noqa`는 **이유를 주석으로 남길 때만** 허용.

> **왜 Ruff인가**: Assist에서 이미 썼다. 학습 비용 0.
> Black + isort + flake8을 하나로 대체하고 압도적으로 빠르다.

### 타입 힌트
- **함수 시그니처에는 붙인다.** 내부 변수는 필요할 때만.
- **mypy/pyright는 도입하지 않는다.** 5일 프로젝트에 타입 체커 설정은 오버헤드.

### 단순함이 최우선
- **클래스 계층 만들지 않는다.** 함수로 짠다.
- 추상화는 **같은 코드가 3번 반복될 때** 한다. 미리 하지 않는다.
- 디자인 패턴 적용 금지.

### 파일 역할 고정
```
src/cli.py         진입점, 출력 포맷팅
src/pipeline.py    Step 1/2/3 오케스트레이션
src/schemas.py     Pydantic 모델만
src/prompts.py     프롬프트 문자열만  ← 로직 섞지 말 것
src/verify.py      quote 원문 검증
```

### LLM 호출
- 모델은 `gpt-4o-mini`. 바꿀 때는 **이유를 커밋 메시지에 남긴다.**
- 파싱 실패 시 **1회만** 재시도. 무한 재시도 금지 (비용).
- 모든 호출의 토큰/지연시간을 기록한다.

### 관측 가능성

각 Step의 **원본 LLM 응답**을 `out/run_<timestamp>.json`에 저장한다.
프롬프트를 고쳤을 때 뭐가 달라졌는지 비교할 수 없으면 튜닝이 불가능하다.

#### `out/`은 커밋한다 ⚠️

**`prompt:` 커밋과 그 프롬프트로 돌린 실행 결과가 같은 커밋에 들어가야 한다.**
그래야 "이 프롬프트 수정이 이 결과를 만들었다"가 증명된다.
`out/`을 gitignore하면 프롬프트 커밋만 남고 결과는 사라진다. 튜닝의 근거가 통째로 없어진다.

단, **입력에 실명·연락처가 없는 샘플로 돌린 것만.**
실제 이력서로 돌린 결과는 `out/private/`로 간다 (gitignore됨).
이건 규율이 아니라 **코드가 강제한다** — `pipeline.resolve_out_dir()`가 입력 경로를 보고 라우팅한다.
`out/run_*.json`에는 이력서 전문이 들어가므로(step2 프롬프트에 통째로 박힌다) 규율에 맡기면 언젠가 한 번은 샌다.

#### 기록 스키마 고정

`out/run_*.json`의 `summary` 블록은 아래 필드를 **반드시** 포함한다. 빼거나 이름을 바꾸면 과거 실행과 비교가 끊긴다. **추가만 한다.**

| 필드 | 이유 |
|---|---|
| `model`, `temperature` | 모델/샘플링을 바꾼 실행을 섞어 보면 결론이 오염된다 |
| **`prompt_hash`** | **`prompts.py` 전체의 해시. 이게 핵심이다** |
| `requirements_count`, `demoted_count` | 강등률의 분모/분자 |
| `latency_s`, `tokens_in`, `tokens_out`, `cost_usd` | 비용/성능 추적 |

> **`prompt_hash`를 두는 이유**
> 이 값이 있으면 "강등률이 낮았던 실행들은 전부 이 프롬프트 버전이었다"를 **코드로 뽑을 수 있다.**
> 없으면 커밋 로그를 손으로 뒤져야 한다. D1에 3줄이면 되는데 D3에 소급하려면 전부 재실행이다.

계측값은 Pydantic `@computed_field`로 노출한다. 순수 `@property`는 `model_dump_json()`에 실리지 않아서
**터미널에는 찍히는데 파일에는 없는** 사고가 난다. 기록되지 않는 계측은 없는 것과 같다.

## 3. Git

### 커밋 메시지 — Conventional Commits

```
<type>: <무엇을 왜 했는지 한 줄>
```

| type | 용도 |
|---|---|
| `feat` | 기능 추가 |
| `fix` | 버그 수정 |
| **`prompt`** | **프롬프트 수정** ← 이 프로젝트 전용. 가장 자주 쓴다 |
| `refactor` | 동작 변화 없는 정리 |
| `chore` | 설정, 의존성, gitignore |
| `docs` | 문서 |

> **`prompt:` 타입을 따로 두는 이유**
> 표준 Conventional Commits엔 없다. 하지만 이 프로젝트에서 가장 잦고 가장 중요한 변경이 프롬프트다.
> `git log --oneline --grep="^prompt:"` 하나로 **프롬프트 진화 과정 전체**가 나온다.

### 메시지에는 반드시 "왜"를 넣는다

```
❌ prompt: update step 2
✅ prompt: Step 2에 "복사만 하라" 제약 추가 - 요약 인용 방지

❌ feat: add verify
✅ feat: quote 원문 대조 검증 추가 - LLM이 근거 문장 지어내는 문제

❌ fix: bug
✅ fix: 요구사항 0개일 때 Step 3 크래시 - 빈 리스트 가드 추가
```

"왜"가 없으면 **나중에 내가 봐도 모르고, 면접관이 봐도 모른다.**

### 커밋 기준 — "자주"가 아니라 "언제"

**되돌아오고 싶은 지점마다 커밋한다.** 시간 기준이 아니라 **상태 기준**이다.

| 커밋 시점 | 예시 |
|---|---|
| **돌아가는 상태가 됐을 때** | Step 1이 처음으로 요구사항을 뱉었다 |
| **프롬프트를 고쳤을 때** ⚠️ | **1회 수정 = 1커밋. 절대 묶지 않는다** |
| **뭔가 지우거나 크게 고치기 직전** | 에이전트에게 리팩터링 시키기 전 |
| **하루 끝** | 무조건 |

> ⚠️ **프롬프트 수정은 반드시 1:1로 커밋한다.**
> D2에 프롬프트를 10번 넘게 고친다. 이걸 한 번에 묶으면
> **어떤 수정이 결과를 좋게 만들었는지 추적이 불가능해진다.**
> `out/run_*.json`과 커밋이 짝이 맞아야 튜닝이 성립한다.

### 커밋 author
- **커밋 author는 본인 계정 하나로 통일한다.**
- **AI 에이전트 co-author trailer(`Co-Authored-By: Claude ...`)는 붙이지 않는다.**
- 도구가 기본값으로 자동으로 붙이려 하므로, 매번 의식적으로 빼야 한다. 명시해두는 것이 유일한 방어다.

### 브랜치
- **`main`에 직접 커밋한다.** 혼자 하는 5일 프로젝트에 브랜치 전략은 오버헤드다.

## 4. AI 에이전트 사용 규칙

### 자동화하면 안 되는 것 ⚠️
- **`prompts.py` 튜닝은 손으로 한다.** 위임 금지.
  - 결과를 눈으로 보고 → 고치고 → 다시 보는 사이클 자체가 이 프로젝트의 핵심 역량이다.
  - "더 좋게 해줘"라고 시키면 뭐가 왜 좋아졌는지 모른 채 끝난다.
- **코어 파이프라인(Step 1/2/3)은 오토파일럿으로 돌리지 않는다.**

### 반드시 할 것
- 코드를 받으면 **그날 안에** 읽는다. 다음 날로 미루지 않는다. (미루면 안 온다)
- 에이전트에게 **"이거 왜 이렇게 짰어?"**를 묻는다. 이해 못 한 코드는 남기지 않는다.

### 설명할 수 있어야 하는 3가지
이것만 이해하면 된다. 나머지(보일러플레이트 등)는 몰라도 된다.

1. **왜 Step 1/2/3을 분리했나** — 한 번에 안 시킨 이유
2. **quote 검증 로직** — LLM이 근거를 지어내는 걸 어떻게 잡는가
3. **뭘 안 만들었고 왜** — 이건 코드가 아니라 결정이다

## 5. 스코프 — 만들지 않는 것

아래는 **의도적으로** 뺐다. 추가하자는 제안이 오면 거절한다.

| 안 만드는 것 | 이유 |
|---|---|
| 이력서 첨삭/재작성 | 결과 품질을 유저가 검증하기 어렵다. 다른 제품이다 |
| 합격 확률 점수 | 근거 없는 숫자는 신뢰를 깎는다 |
| 로그인/회원가입/저장 | 5일 안엔 핵심 가치 검증이 우선 |
| 공고 URL 크롤링 | 사이트마다 구조가 달라 시간 블랙홀 |
| PDF 파싱 | 텍스트 붙여넣기로 충분 |
| 테스트 코드 | D2 이후 필요하면. **단 `verify.py`는 예외 (아래)** |
| 타입 체커(mypy) | 5일 프로젝트에 설정 오버헤드 |

**"이거 있으면 좋을 것 같은데"는 전부 거절한다.**

### 유일한 예외: `verify.py`의 경계 케이스

`scripts/check_verify.py` — 테스트 프레임워크가 아니라 **함수 하나**다. `uv run python scripts/check_verify.py`.

| 케이스 | 기대 |
|---|---|
| 원문 그대로 | 통과 |
| 개행/공백만 다름 | 통과 |
| LLM이 요약한 문장 (원문에 없음) | **강등** |
| 원문의 절반만 (부분 인용) | 통과 + **경고** |

이 4개가 `normalize`의 경계 그 자체다. README는 "느슨함과 엄격함 사이의 가장 좁은 지점을 잡았다"고 주장하는데, **그 주장을 뒷받침하는 것이 이 파일뿐이다.**
그리고 면접에서 "느슨하면 왜 안 되죠?"를 물었을 때, 케이스를 보여주는 것이 설명하는 것보다 10배 강하다.

**강등은 오직 "원문에 없다"(= 지어냈다)에만 쓴다.** 부분 인용은 경고만 한다 — 원문에 실재하는 이상 지어낸 것이 아니고, 진짜 근거를 강등해서 버리는 것은 갭을 숨기는 것만큼 나쁘다.
